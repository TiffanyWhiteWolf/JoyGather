# "趣聚"平台 团队分工方案

团队规模：10 人，每人负责所属功能域的前端 + 后端全栈开发

> **结构调整说明**：P1 与 P2 合并为新的 P1（账户+个人资料+商家），释放出的 P2 专门负责备选故事，同时从 P6/P8 移出部分故事给 P4/P7，使核心工作量更均衡。

## 详细分工

| 用户故事 | 故事点 | 优先级 | 工作内容 |

## 协作依赖关系

```
P1(账户与个人资料) ──┬── 用户数据（内部消化）
                      │
P3(活动创建) ────────┼── 活动数据 ──→ P4(审核+小队治理)
                      │               └── P5(活动发现/详情)
                      │                   └── P6(地图/报名)
                      │                       └── P7(签到/评价/队列)
                      │
P8(好友与小队) ──────┤
P9(即时通讯) ────────┤
                      │
P10(后台运营) ───────┘

P2(备选) ──→ 依赖各模块能力：P4(AI) → B02, P9(IM) → C05, P3(活动) → C06, P7(存储) → C07
```

依赖建议：
- **P1 先输出用户鉴权机制** → 全团队依赖（所有接口需 Token 鉴权）
- **P3 先输出活动表结构和创建接口** → P4/P5/P6/P7 依赖
- **P9 先输出管理员账号体系** → P10 后台功能依赖
- **P4 先完成人工审核** → 迭代 2 在其基础上叠加 AI 审核
- **P2 的备选故事**需等前序依赖（P4 AI、P9 IM、P3活动、P7存储）就绪后再推进
