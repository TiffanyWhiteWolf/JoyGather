# 一、分析方法说明

## 1.1 用户故事格式

统一采用"角色—诉求—价值"三段式描述：
作为 [用户角色]，我希望 [完成某项功能]，以便 [获得某种价值]。

每条用户故事附带可测试的验收标准。

## 1.2 估算方法（故事点）

采用相对估算的故事点，取斐波那契数列 1、2、3、5、8、13 作为可选值。估算综合考虑业务复杂度、技术风险和不确定性，不等同于人天。

| 故事点 | 典型场景 |
|--------|---------|
| 1 | 配置类、单一查询或极简页面 |
| 2 | 单一流程，少量校验规则 |
| 3 | 标准业务流程，包含基础校验和状态变化 |
| 5 | 跨页面或跨服务流程，含多种状态和异常分支 |
| 8 | 复杂业务闭环，涉及第三方能力或多端同步 |
| 13 | 大型能力，需拆分交付或存在较高技术不确定性 |

## 1.3 优先级定义（MoSCoW 方法）

| 等级 | 含义 | 判定原则 |
|------|------|---------|
| Must-have | 必须有 | MVP 核心闭环不可或缺，或需求文档明确要求必备 |
| Should-have | 应该有 | 重要功能，显著提升体验或运营效率，可在核心闭环后交付 |
| Could-have | 可以有 | 增强型或智能化功能，资源允许时实现 |
| Won't-have | 本次无 | 本期不做，作为后续规划占位 |

# 二、用户故事划分与验收标准

## 模块 A：用户与身份管理

### US-A01 用户注册与邮箱激活

描述：作为访客，我希望使用邮箱和密码完成注册并通过邮件激活账号，以便安全进入平台。

- 故事点：5
- 优先级：Must-have

验收标准：
1. 用户填写邮箱、密码和确认密码后提交注册，系统创建未激活账号并发送激活链接
2. 用户点击有效激活链接后账号激活，可正常登录
3. 邮箱格式不合法、密码长度不足或两次密码不一致时，系统给出明确错误提示
4. 同一邮箱不可重复注册
5. 未激活账号不能登录或使用受限功能

### US-A02 用户登录与状态校验

描述：作为已注册用户，我希望通过邮箱和密码登录，并根据账号状态获得正确的访问权限。

- 故事点：2
- 优先级：Must-have

验收标准：
1. 输入正确邮箱和密码后登录成功，跳转至首页
2. 输入错误时提示"邮箱或密码错误"
3. 账号未激活时提示"请先激活账号"
4. 账号被封禁时提示"账号已被封禁"并显示封禁原因和期限
5. 登录成功后 Token/Session 有效期内无需重复登录

### US-A03 个人资料管理

描述：作为个人用户，我希望维护头像、昵称、性别、生日、个性签名和兴趣标签等资料，以便展示个人形象并获得更匹配的活动推荐。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 可上传或更换头像，支持常见图片格式
2. 昵称全平台唯一，修改时系统实时校验是否被占用
3. 兴趣标签可从平台预设标签中多选
4. 所有资料修改成功后即时生效并在个人主页展示
5. 必填项缺失或格式不合法时拒绝保存并提示

### US-A04 商家认证与资料管理

描述：作为商家用户，我希望上传营业执照或营业凭证申请商家身份，经后台审核通过后可维护商家名称、昵称和关注领域，以便以商家身份运营活动。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 商家用户提交营业执照或营业凭证后，申请进入待审核状态
2. 管理员审核通过后用户获得商家身份标识
3. 审核驳回时显示驳回原因，用户可重新提交
4. 商家资料包括商家名称、商家昵称和关注活动领域
5. 商家资料与个人资料分开展示

## 模块 B：活动全生命周期管理

### US-B01a 基础活动创建

描述：作为活动发起人，我希望填写活动名称、标签、简介、时间、地点（手动输入）、人数上限和报名截止时间等基本信息来创建活动，以便发布线下活动供他人发现和报名。

- 故事点：5
- 优先级：Must-have

验收标准：
1. 必填字段包括活动名称、标签、简介、时间、地点、人数上限和报名截止时间
2. 地点支持手动输入地址文本
3. 开始时间须晚于当前时间，报名截止时间须不晚于开始时间
4. 人数上限须为正整数
5. 必填字段缺失或规则校验不通过时，系统阻止提交并逐项提示

### US-B01b 地图选点扩展

描述：作为活动发起人，我希望在创建活动时通过地图选点确定活动地点，以便活动位置展示更精确。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 活动创建页提供地图选点入口
2. 用户在地图中搜索或标记点位后，系统自动填充地址和经纬度坐标
3. 选点结果可在活动详情页正确展示
4. 地图服务不可用时支持回退到手动输入地址

### US-B02 AI 活动策划

描述：作为活动发起人，我希望输入活动主题或选择活动类型后由 AI 自动生成活动基础信息和参与设置，以便快速完成活动策划。

- 故事点：8
- 优先级：Could-have

验收标准：
1. 用户输入活动主题或选定活动类型后，可触发 AI 生成
2. AI 生成内容至少包含活动名称、简介、建议时间和人数上限
3. 生成结果以可编辑表单展示，用户可逐项修改
4. AI 服务超时或失败时，系统提示异常并允许用户手动创建
5. AI 生成的内容提交后仍需进入正常审核流程

### US-B03 活动模板与克隆

描述：作为活动发起人，我希望使用平台提供的活动模板或克隆已发布活动，以便减少重复填写工作。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 平台提供至少 6 类活动模板：运动健身、户外徒步、桌游聚会、学习交流、公益活动、城市探索
2. 选择模板后系统自动预填活动基础信息，用户可修改后提交
3. 发起人可对已发布或已完成活动执行克隆，系统复制基础信息和参与设置
4. 克隆生成的新活动以草稿形式保存，用户修改差异字段后提交
5. 克隆不复制报名数据、签到记录和评价数据

### US-B04 草稿保存与提交

描述：作为活动发起人，我希望在创建活动过程中随时保存草稿，后期继续编辑并最终提交审核，以便灵活安排活动创建流程。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 创建过程中支持随时保存草稿，保存时不强制校验必填字段
2. 草稿可在"我的草稿"列表中查看、继续编辑或删除
3. 草稿不会出现在首页、搜索或地图发现中
4. 提交时系统校验必填字段和业务规则，通过后进入审核流程
5. 重复点击提交不会创建重复审核任务

### US-B05 AI 自动内容安全审核

描述：作为平台运营方，我希望在活动提交后自动进行 AI 内容安全审核，检测违法违规、低俗和危险内容，以便保障平台内容安全。

- 故事点：5
- 优先级：Must-have

验收标准：
1. 活动提交后系统自动调用 AI 审核，覆盖活动名称、简介、标签等文本内容
2. AI 判定低风险的普通活动可直接发布
3. AI 判定存在风险或无法确定的活动转入人工审核队列
4. AI 服务异常时活动应转入人工审核或保持在待处理状态，不得默认放行
5. 审核结论和风险标签记录留存备查

### US-B06 人工审核

描述：作为审核管理员，我希望对报名人数超过 50 人或 AI 审核有疑问的活动进行人工审核，并给出通过、驳回或要求修改的处理结果。

- 故事点：5
- 优先级：Must-have

验收标准：
1. 报名人数上限超过 50 人的活动自动进入人工审核队列
2. AI 审核标记异常的活动转入人工审核队列
3. 审核员可查看活动完整信息和风险提示
4. 审核结果包含通过、驳回、要求修改三种
5. 驳回或要求修改时必须填写原因并通知活动发起人
6. 所有审核操作保留操作人、时间、结果和原因记录

### US-B07a 首页最新信息流

描述：作为用户，我希望在首页看到按时间排序的最新活动列表，以便快速了解平台新增的活动。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 首页提供"最新"Tab，活动按发布时间倒序排列
2. 活动卡片展示名称、时间、地点、名额状态和标签
3. 未发布、草稿、已下架活动不展示在信息流中
4. 点击活动卡片进入活动详情页
5. 支持下拉刷新加载更多活动

### US-B07b 首页推荐信息流

描述：作为用户，我希望在首页看到基于我的兴趣标签推荐的活动，以便发现更适合我的活动。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 首页提供"推荐"Tab，活动基于用户兴趣标签排序
2. 推荐结果随用户兴趣标签变化而更新
3. 用户未设置兴趣标签时展示默认推荐列表（如热门活动）
4. 活动卡片展示方式与最新 Tab 保持一致

### US-B07c 附近发现

描述：作为用户，我希望在首页查看当前位置附近的活动列表，并可通过地图模式直观查看活动分布。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 首页提供"附近"Tab，按距离展示活动
2. 未授权定位时提示授权或允许手动选择城市
3. 活动列表与最新、推荐 Tab 保持一致的卡片样式

### US-B08a 关键词搜索

描述：作为用户，我希望通过关键词搜索活动名称和标签，以便快速找到特定主题的活动。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 搜索页提供搜索输入框，用户输入关键词后搜索
2. 搜索结果匹配活动名称、标签和简介
3. 无结果时展示空状态并推荐调整关键词
4. 搜索结果可点击进入活动详情页

### US-B08b 高级筛选

描述：作为用户，我希望按活动类型、时间范围、城市、费用和距离范围进行组合筛选，以便精确缩小活动范围。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 搜索页和首页提供筛选器入口
2. 筛选条件包括活动类型（多选）、时间范围、城市、费用和距离范围
3. 多个筛选条件可组合使用，结果实时更新
4. 清空筛选条件后恢复默认结果

### US-B09 地图模式

描述：作为用户，我希望在地图上以点位形式查看活动位置，以便直观了解周边活动分布。

- 故事点：8
- 优先级：Must-have

验收标准：
1. 附近 Tab 和搜索页均提供地图模式入口（需求明确：地图模式必须有）
2. 地图以标记点形式展示当前范围内已发布的活动
3. 拖动或缩放地图时，系统刷新当前区域的活动点位
4. 点击标记点弹出活动摘要卡片，可跳转至详情页
5. 地图服务异常时提供明确降级提示

### US-B10 活动详情页

描述：作为用户，我希望在活动详情页查看完整信息和实时状态，以便决定是否报名参与。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 详情页展示活动名称、标签、简介、时间、地点、人数上限/已报名人数、发起人信息和安全须知
2. 活动状态正确显示：未开始、报名中、截止、活动中、已结束
3. 状态随时间和业务动作自动变化
4. 不同状态下展示对应的操作入口（报名中显示"立即报名"，截止后隐藏报名按钮）
5. 已下架或不存在的活动提示不可访问

### US-B11 报名校验与确认

描述：作为用户，我希望点击"立即报名"后系统完成名额、信誉和年龄等校验，并在通过后弹窗确认并填写必要信息，以便顺利完成报名。

- 故事点：5
- 优先级：Must-have

验收标准：
1. 点击"立即报名"后系统校验：名额是否满额、用户信誉、年龄等条件
2. 校验通过后弹出确认窗口，展示安全须知摘要
3. 根据活动要求填写必要的报名信息
4. 用户确认后报名成功，系统扣减名额
5. 校验失败时逐项告知不通过原因
6. 并发报名最后名额时保证原子性，不超卖

### US-B12 取消报名

描述：作为已报名用户，我希望在报名截止时间前取消报名，以便释放名额给其他用户。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 已报名用户可在报名截止前发起取消，需二次确认
2. 取消成功后系统释放名额
3. 存在等待队列时自动触发第一位用户递补
4. 报名截止后不再允许取消

### US-B13 等待队列

描述：作为用户，我希望在活动名额已满时进入等待队列，并在名额释放时收到通知和限时确认机会，以便仍有机会参与活动。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 名额已满时用户可选择加入等待队列，并显示当前等待人数
2. 同一用户不可重复入队
3. 名额释放后系统自动通知队列第一位用户
4. 被通知用户须在限定时间内确认报名
5. 超时未确认则顺延至下一位用户

### US-B14 扫码签到与签到管理

描述：作为活动发起人，我希望生成签到二维码并在活动现场进行扫码签到（可结合位置校验），同时能够查看报名用户的签到状态，以便确认到场真实性和管理现场。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 发起人在活动开始前可生成与活动绑定的签到二维码
2. 已报名参与者扫码后完成签到，记录签到时间
3. 未报名用户或使用无效二维码时拒绝签到
4. 可启用位置校验，判断用户是否在活动地点附近
5. 发起人在活动管理页可查看报名列表和签到状态
6. 活动结束后签到数据进入统计模块，重复扫码不重复计数

### US-B15 活动图文总结

描述：作为活动发起人，我希望在活动结束后发布图文总结，AI 可辅助对图片进行分类（合影、场地、过程记录等），经人工确认后生成最终总结，以便沉淀活动成果。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 活动结束后发起人可上传图片并编辑文字发布图文总结
2. AI 可将图片分类识别为合影、场地、过程记录、物资、成果展示等类别
3. AI 分类结果须由发起人逐项确认或调整后才可生成最终总结
4. AI 服务异常时不影响手动整理和发布
5. 未结束活动不能发布正式总结

### US-B16 用户评价

描述：作为活动参与者，我希望在活动结束后一定期限内对活动进行评价，以便反馈活动体验。

- 故事点：2
- 优先级：Should-have

验收标准：
1. 活动结束后参与者可在有效期内提交评价
2. 评价入口在期限到达后自动消失
3. 未参加活动的用户不可评价
4. 重复评价按平台规则处理

## 模块 C：好友社群

### US-C01 添加好友

描述：作为用户，我希望通过个人主页、活动参与者列表或个人二维码发起好友申请，对方可同意或拒绝，以便建立社交关系。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 可从他人主页、活动参与者列表或扫描个人二维码发送好友申请
2. 对方收到申请后可选择同意或拒绝
3. 已是好友或处于黑名单关系时不可重复发送申请
4. 申请处理后关系状态正确更新

### US-C02 好友管理与关注关系

描述：作为用户，我希望为好友设置备注和分组标签，并可单向关注其他用户，互相关注后自动升级为好友，以便管理社交关系。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 可为好友设置备注名称和分组标签（仅本人可见）
2. 删除好友后双方好友关系自动解除，备注和分组信息失效
3. 用户可单向关注其他用户
4. 双方互相关注后关系自动升级为好友
5. 任一方取消关注后，互关和好友状态同步解除

### US-C03 兴趣小队建立

描述：作为用户，我希望建立兴趣小队时填写小队名称、兴趣标签、加入方式（公开或审核）和人数上限，创建者自动成为队长，以便聚集同好。

- 故事点：5
- 优先级：Should-have

验收标准：
1. 创建时须填写小队名称、兴趣标签、加入方式和人数上限
2. 加入方式支持公开加入和审核加入
3. 创建者自动成为队长
4. 创建成功后自动生成小队群聊

### US-C04 兴趣小队发现与加入

描述：作为用户，我希望按小队名称或标签查找并申请加入小队，以便参与同好群体。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 可按小队名称或兴趣标签搜索小队
2. 公开小队可直接加入
3. 审核小队须队长或管理员同意后方可加入
4. 人数已满、已解散或存在黑名单关系时拒绝加入并提示原因

### US-C05 小队群聊与协作

描述：作为小队成员，我希望小队自动生成群聊，成员可发布公告、使用@提醒、参与投票、上传文件，以便高效协作。

- 故事点：8
- 优先级：Could-have

验收标准：
1. 小队创建成功后自动生成群聊，成员加入/退出时同步
2. 队长和管理员可发布公告和使用"@所有人"
3. 成员可使用"@"提醒指定成员
4. 成员可参与群投票
5. 成员可上传和查看群文件
6. 已解散小队的群聊停止发送新消息

### US-C06 队内活动

描述：作为队长或管理员，我希望发布仅小队成员可见的队内活动，成员可查看和报名，以便组织小队内部活动。

- 故事点：5
- 优先级：Could-have

验收标准：
1. 仅队长和管理员可发布队内活动
2. 小队成员可查看和报名队内活动
3. 非小队成员不能访问队内活动
4. 小队停用或解散后不可新增队内活动

### US-C07 小队相册与积分榜

描述：作为小队成员，我希望上传活动照片到小队相册并查看积分榜，以便沉淀活动记忆和了解活跃排名。

- 故事点：5
- 优先级：Could-have

验收标准：
1. 成员可上传照片到小队相册
2. 队长和管理员可删除相册内容
3. 系统根据参与小队活动、发布动态等行为计算积分
4. 积分榜展示成员昵称、积分和名次

### US-C08 小队权限管理

描述：作为小队队长，我希望设置队长、管理员和普通成员三类角色并分配相应权限，以便安全治理小队。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 队长可管理小队信息、成员、管理员、公告、活动、文件和相册，并可解散小队
2. 管理员可审核成员、管理公告、活动、文件和相册，但不能解散小队
3. 普通成员可参与群聊、投票、上传文件/照片、报名活动和查看积分榜
4. 权限校验在前端和后端均执行

### US-C09 小队解散

描述：作为队长，我希望在必要时解散小队，以便停止不再运营的小队。

- 故事点：2
- 优先级：Should-have

验收标准：
1. 仅队长可执行解散操作，需二次确认
2. 解散后所有成员自动退出
3. 群聊和队内活动停止使用
4. 小队不再出现在发现列表中

### US-C10 即时通讯

描述：作为用户，我希望与好友或小队成员之间发送文字、表情、图片和位置消息，并支持已读状态、撤回和转发，以便高效沟通。

- 故事点：8
- 优先级：Should-have

验收标准：
1. 好友之间和小队群聊中支持发送文字、表情、图片和位置共享消息
2. 消息发送后显示未读或已读状态
3. 用户可在发送后 2 分钟内撤回本人消息
4. 消息可转发至其他好友或小队群聊
5. 非好友且非同队关系不可发送消息

## 模块 D：后台管理

### US-D01 管理员登录

描述：作为平台管理员，我希望使用系统预创建的账号登录后台并修改本人密码，以便安全进入运营管理系统。

- 故事点：1
- 优先级：Must-have

验收标准：
1. 管理员账号由系统预创建，不提供注册入口
2. 管理员使用账号密码登录后台
3. 管理员可修改本人登录密码
4. 登录失败时给出通用提示，不泄露账号状态

### US-D02 用户查询

描述：作为管理员，我希望查询个人用户和商家用户，查看基本信息、账号状态以及发布的活动和创建的小队，以便支持运营管理。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 可按昵称、邮箱或用户类型查询用户
2. 可查看用户基本信息、账号状态、发布的活动列表和创建的小队列表
3. 管理员不可修改用户个人资料和密码

### US-D03 用户封禁与解封

描述：作为管理员，我希望对有违规行为的用户执行封禁（填写原因和期限）并在问题处理后解封，以便维护平台秩序。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 封禁用户时须填写原因并选择封禁期限
2. 封禁后用户无法登录平台
3. 解封或期限到达后用户恢复正常使用
4. 原有账号数据（活动、好友、小队等）不受影响
5. 所有操作保留操作人、时间、原因记录

### US-D04 商家申请审核

描述：作为管理员，我希望审核商家用户提交的营业执照申请，通过或驳回，以便控制商家入驻质量。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 管理员可查看待审核商家申请的资质材料
2. 可选择通过或驳回
3. 驳回时必须填写原因
4. 审核通过后用户获得商家身份标识

### US-D05 活动管理（后台）

描述：作为管理员，我希望查询平台全部活动并按状态筛选，对违规活动执行下架（须填写原因）和恢复操作，以便治理违规内容。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 可按活动名称、状态、发起人搜索和筛选活动
2. 可查看活动完整信息
3. 下架已发布违规活动时须填写原因
4. 下架后活动不再对用户展示和开放报名
5. 问题处理后可恢复活动

### US-D06 小队管理（后台）

描述：作为管理员，我希望查看平台全部小队及其成员、活动和举报记录，对违规小队执行停用（须填写原因）和恢复，以便维护社群秩序。

- 故事点：3
- 优先级：Should-have

验收标准：
1. 可按小队名称或队长搜索小队
2. 可查看小队基本信息、成员列表、队内活动和举报记录
3. 管理员不可直接修改小队名称、标签、公告和成员身份
4. 停用时须填写原因，停用后不可新增成员、活动或内容
5. 原有成员可查看历史信息
6. 问题处理后管理员可恢复小队正常功能

## 模块 E：非功能性需求

### US-E01 页面加载性能

描述：作为用户，我希望 App 和 Web 端在正常网络下 5 秒内完成启动并进入可操作页面，以便获得流畅体验。

- 故事点：3
- 优先级：Must-have

验收标准：
1. App 从完全关闭状态启动至登录页或首页可正常操作的时间小于 5 秒
2. Web 端从进入系统至登录页或首页可正常操作的时间小于 5 秒
3. 性能测试需明确记录设备、网络和浏览器信息

### US-E02 后台接口性能

描述：作为系统，我希望每个后台业务接口在排除第三方依赖后单次响应时间不超过 2000 毫秒，以便保证操作流畅。

- 故事点：3
- 优先级：Must-have

验收标准：
1. 不涉及第三方服务的后台业务接口单次响应时间小于 2000 毫秒
2. AI 服务、地图、短信等第三方接口及其调用方接口不纳入此指标
3. 建议以 P95 作为上线门槛

### US-E03 第三方服务降级

描述：作为用户，我希望在 AI、地图或邮件服务异常时获得清晰的反馈和替代路径，以便正常完成核心操作。

- 故事点：5
- 优先级：Should-have

验收标准：
1. AI 服务失败时提示用户并可手动继续操作
2. 地图服务失败时提供手动输入地址的降级方案
3. 邮件发送失败时记录失败日志并支持重新发送
